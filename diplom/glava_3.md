# 3 ПРОЕКТИРОВАНИЕ И РЕАЛИЗАЦИЯ ПРОГРАММНОГО КОМПЛЕКСА

## 3.1 Логическая структура комплекса и конфигурирование

**Стек технологий.** Реализация опирается на **C\#** и **.NET 8** [9]: статическая типизация, предсказуемая сборка, зависимости через **NuGet**. Консольный запуск без GUI соответствует сценарию **пакетной** обработки всех книг **.xlsx** в рабочем каталоге. Для чтения таблиц используется **ClosedXML** [8] (формат Office Open XML без установленного Excel). **Baseline** для многомерного сравнения (линейные и полиномиальные модели) считаются средствами **Math.NET Numerics**, в том числе через устойчивые процедуры линейной алгебры (**QR** и связанные вычисления) [7]. Конфигурация эволюции и пороги контрольных файлов хранятся в **JSON** и десериализуются **System.Text.Json** [9] с учётом имён свойств в нотации **camelCase**, принятой в файле **`gp_settings.json`**.

**Модульная структура.** Вся предметная логика сосредоточена в пространстве имён **`SymbolicRegression`** в том же модуле, что и точка входа **`Program`** (**`Program.cs`** проекта **`kursovaia`**). Такое решение упрощает сопровождение курсового/дипломного прототипа: один файл компилируется в одну сборку, зависимости перечислены в **`.csproj`**, а запуск из IDE по умолчанию использует рабочий каталог с конфигурацией и таблицами.

**Загрузка конфигурации.** При старте **`Main`** вызывает **`LoadRuntimeConfig()`**: если **`gp_settings.json`** отсутствует в **`Environment.CurrentDirectory`**, берутся **встроенные** значения порогов **`benchmarkThresholds`** и пустой словарь профилей; если файл есть, он десериализуется в **`AppRuntimeConfig`** с полями **`gpProfiles`** и **`benchmarkThresholds`**. Ключи профилей и имён эталонных файлов сравниваются **без учёта регистра** (**`StringComparer.OrdinalIgnoreCase`**), что снижает риск опечаток при ручном редактировании JSON.

**Переопределение параметров GP.** Класс **`GpProfileOverride`** задаёт **неполный** набор полей: в JSON указывают только те параметры, которые должны заменить базовые значения для данного имени профиля. Поддерживаются, в частности: **`maxDepth`**, **`populationSize`**, **`generations`**, **`restarts`**, **`mutationRate`**, **`crossoverRate`**, **`terminalConstProbability`**, **`complexityPenalty`**, **`invalidPredictionPenalty`**, **`maxExpArg`**, **`maxPowAbs`**, массив **`allowedOps`**. Метод **`SymbolicRegressionEngine.SetExternalProfileOverrides`** копирует словарь в статическое хранилище; при построении **`GpConfig`** для конкретного **`GrammarProfile`** вызывается **`ApplyOverride`**, так что JSON-правки меняют поведение эволюции **без перекомпиляции**.

**Связь имён профилей с внутренней грамматикой.** В коде перечисление **`GrammarProfile`** (Linear, Polynomial, Trigonometric, General, Superposition) отображается на строковые ключи JSON: **`"linear"`**, **`"polynomial"`**, **`"trigonometric"`**, **`"general"`**, **`"superposition"`**. Для одномерного случая профиль выбирается **эвристически** после **`AnalyzeDataPattern()`**; для многомерного с **`base_functions.txt`** — **`Superposition`**, без успешно загруженных **φ** — **`General`**. Пример фрагмента JSON с ключом **`gpProfiles`** и численными значениями удобно вынести в **приложение** (в репозитории — **`kursovaia/kursovaia/gp_settings.json`**). Какой **конструктор** **`SymbolicRegressionEngine`** вызывается из разных мест пакета (**Main**, **`ProcessMultipleRegression`**, ветвь log-fit), показано на **рис. 3.7** (файл **`diplom/diagrams/fig_3_7_engine_constructors.drawio`**).

**Файл `base_functions.txt`.** При **d ≥ 2** факторах (столбцы до последнего) и отклике в **последнем** столбце программа может работать в режиме **суперпозиции** (п. 1.2): в листья дерева наряду с переменными **x₁, …, x_d** допускаются терминалы **`base`**, указывающие на заранее разобранные выражения **φ₁(X), …, φ_K(X)**. Файл ищется по пути **`Path.Combine(Environment.CurrentDirectory, "base_functions.txt")`**. Если файла нет, **`LoadBaseFunctions`** возвращает **пустой** список — прогон продолжается в **общей** грамматике без суперпозиции.

**Разбор и устойчивость к ошибкам.** Строки с **`#`** в начале (после пробелов) и пустые строки пропускаются. Допустимы форматы **`имя = выражение`** и одно **`выражение`** без имени; в парсер уходит правая часть после **`=`**. Для каждой строки вычисляется максимальный индекс **`x_j`** в тексте; если **j** больше числа факторов в таблице, строка **отбрасывается**, предупреждение может быть собрано в список **`warnings`**. Ошибка разбора **одной** строки **не** прерывает загрузку остальных. В конструкторе **`SymbolicRegressionEngine(double[][], …)`** флаг **`superpositionMode`** становится истинным **только** если список базовых деревьев **непуст**.

**Мини-язык выражений** (рекурсивный спуск в **`ExpressionParser`**): переменные **`x1`**, **`x2`**, … (индекс не выше **`varCount`**); числа с **точкой** или **запятой** (запятая нормализуется к точке); операции **`+ − * /`**, возведение **`^`** (**правоассоциативно**, в дерево попадает операция **`pow`**); скобки; унарный минус (**операция `neg`**); функции **`sin`**, **`cos`**, **`exp`**, **`log`** с обязательными скобками вокруг аргумента. Иные идентификаторы приводят к **`FormatException`** на уровне строки. Для предметной настройки в файл помещают осмысленный минимальный набор **φ**, остальное комментируют **`#`**. Несколько версий набора удобно хранить под разными именами и перед прогоном переименовывать нужную в **`base_functions.txt`**.

**Ограничения на вычисление дерева.** При интерпретации узлов в **`Evaluate`** действуют правила, снижающие числовой «взрыв» и бессмысленные значения: деление при **|b| < 10⁻¹⁰** даёт **NaN**; **`log`** при аргументе **≤ 0** — **NaN**; **`exp`** при аргументе **> maxExpArg** возвращает **`+∞`** (что затем учитывается как невалидное предсказание); для **`pow`** показатель по модулю ограничен **`maxPowAbs`**, при **a < 0** и нецелом **p** результат **NaN**. Эти пороги совмещаются с штрафом **`invalidPredictionPenalty`** в целевой функции (п. 3.3). Ветвление по типу узла при вычислении дерева изображено на **рис. 3.8** (**`diplom/diagrams/fig_3_8_evaluate_tree.drawio`**).

## 3.2 Алгоритм обработки файлов и режимы работы

**Точка входа и обход каталога.** **`Main`** после загрузки конфигурации получает массив путей **`Directory.GetFiles(..., "*.xlsx")`**, исключает имена, начинающиеся с **`~$`**, сортирует по имени файла (лексикографически, без учёта регистра) и для каждой книги пишет блок в **`result.txt`**. Пустой каталог даёт короткое сообщение в протоколе. Исключения при обработке **одного** файла перехватываются: в протокол выводится **`ex.Message`**, цикл переходит к следующей книге. В конце вызывается **`WriteBenchmarkSummary`** — сводка **PASS/FAIL** по тем файлам, для которых в JSON заданы пороги, и по которым удалось вычислить **RMSE** и размах **y**. Обобщённая блок-схема этой логики (цикл по файлам, ветвление **> 2** столбцов / одномерный случай, сводка порогов) приведена на **рис. 3.1** (**`diplom/diagrams/fig_3_1_main_flow.drawio`**); концептуально близкий поток «ввод → ядро → отчёт» может дублироваться **рис. 2.3** (гл. 2).

**Выбор листа и столбцов.** Открывается **`workbook.Worksheet(1)`** — **первый лист**. Число задействованных столбцов определяется по **`LastColumnUsed()`**; если столбцов **больше двух**, включается ветвь **`ProcessMultipleRegression`**; **ровно два** числовых столбца далее интерпретируются как пары **(x, y)** в одномерном сценарии (**важно:** при **двух** столбцах многомерная ветка **не** используется).

**Одномерные данные (два столбца).** Строки читаются сверху вниз. Первая строка считается **заголовком**, если **обе** ячейки **нельзя** привести к **`double`**; иначе она трактуется как данные. Дальше собираются только строки, где **оба** числа успешно разобраны; неполные строки **молча** отбрасываются. При **n ≤ 1** выводится диагностика недостатка данных. Для наглядности в протокол печатаются **первые 10** пар **(x, y)** и многоточие при большей длине. Детализация одномерной ветви **`Main`**: чтение, **`SymbolicRegressionEngine`**, **`Evolve`**, сравнение с **`exp(g(x))`** — на **рис. 3.5** (**`diplom/diagrams/fig_3_5_main_1d_branch.drawio`**).

**Эвристика типа ряда и эволюция 1D.** Создаётся **`SymbolicRegressionEngine(xArr, yArr)`**; внутри вызывается **`AnalyzeDataPattern()`** (линейность, признаки тригонометрии, полинома, экспоненты и т.д.). Пользователю в протокол выводится строка **`Определенный тип данных: …`** (**`GetDataPattern()`**). Затем вызывается **`Evolve()`** (п. 3.3): отдельно отметим, что в протоколе числовая строка **`Оценка (BIC): …`** фактически выводит **целевую функцию обучения** (штрафованную MSE на **обучающей** части выбранного рестарта), а не классический байесовский критерий — подпись в коде историческая, для записки важно не смешивать понятия с информационными критериями из п. 1.3. Последовательность инициализации одномерного движка — **рис. 3.3** (**`diplom/diagrams/fig_3_3_engine_init_1d.drawio`**); дерево решений **`AnalyzeDataPattern()`** — **рис. 3.4** (**`diplom/diagrams/fig_3_4_analyze_data_pattern.drawio`**).

**Ветвь `exp(g(x))`.** Если все **y > 0** и точек **не меньше пяти**, дополнительно строится вторая эволюция на парах **(x, ln y)** с **принудительным** профилем **`GrammarProfile.General`**, чтобы в логарифмическом пространстве были доступны все операции. Полученное дерево **g** оборачивается в узел **`exp`**, **RMSE** проверяется на исходных **y**. Улучшение принимается, если **`logRmse < 0.95 · bestRmse`** от прямого запуска — типичный приём, чтобы не заменять устойчивый фит на шумное улучшение. В протокол выводятся подписи **`direct`** / **`log-fit`**.

**Многомерные данные (≥ 3 столбцов).** Первая строка листа **всегда пропускается** как заголовок (**`startRow = 2`**): при отсутствии текстового заголовка **первая строка данных теряется**, это **соглашение формата**, о котором пользователя стоит предупреждать. Строки собираются в список векторов; нечисловая ячейка в строке делает её **целиком** невалидной. Матрица **X** и вектор **y** формируются по правилу «последний столбец — отклик».

**Hold-out 20 % и кандидаты.** Набор перемешивается генератором **`Random(123)`**; **`testCount = max(1, ⌊n/5⌋)`**, остальное — обучение. Для обучения запускается GP на **`XTrain`**, **`yTrain`** с подгрузкой **`base_functions.txt`**. Параллельно строятся **baseline-деревья** (аналитические конструкции без полноценного GP): **линейная**, **квадратичная** (по признакам), специализированный шаблон **`sin2-x2exp`**, **log-линейная** — в коде они добавляются в список кандидатов с **RMSE** на train и test. Так достигается **устойчивость**: если GP застрянет в плохом локальном минимуме, простая структура может выиграть на **тесте**. Блок-схема **`ProcessMultipleRegression`** — **рис. 3.6** (**`diplom/diagrams/fig_3_6_process_multiple_regression.drawio`**); этот **hold-out** не следует путать с разбиением **train/val** внутри **`Evolve()`** (**рис. 3.2**).

**Выбор итоговой модели.** Среди кандидатов с конечным **`RMSE_test`** выбирается **минимальный тестовый RMSE**; при равенстве задан **приоритет проще**: **`linear` → `sin2-x2exp` → `quadratic` → `log-linear` → `GP`**. Для победителя печатается развёрнутая формула (**`ExpandAndSimplifyBaseFormula`**) и метрики: **RMSE** на train/test, **R²** на обучении, относительный **RMSE/yRange** на тесте, `overfit_gap`, статус **`OK`/`WARNING`** при слишком большом отрыве теста от обучения.

**Контроль эталонов.** **`WriteBenchmarkSummary`** для каждого ключа в **`benchmarkThresholds`** сопоставляет имя файла **без пути**; сравнивает **RMSE** (в многомерном сценарии это **тестовый** RMSE выбранной модели) с порогом; дополнительно вычисляется **относительная** ошибка **`RMSE / yRange`** и проверка **`rel ≤ 0.005`** (константа **`RelativeRmsePassThreshold`** в **`Program`**) для пометки относительного прохождения. Итог — компактная таблица в конце **`result.txt`**.

## 3.3 Основные классы и сценарии эволюционного поиска

**`ExpressionNode`.** Узел хранит строковую **`Operation`**, список потомков **`Children`**, опционально **`Constant`** (лист с операцией **`const`**), **`VariableIndex`** (нумерация с нуля: **`x1`** → 0), **`BaseFunctionIndex`** (ссылка на элемент списка **φ**), **`Depth`** (служебная глубина при генерации). Методы **`Clone()`** и **`ToString()`** поддерживают эволюцию и человекочитаемый вывод; лист **`base`** в **`ToString`** отображается как **`f1`**, **`f2`**, …. Наглядный пример дерева и подписи листьев — **рис. 1.1** в гл. 1 (**`diplom/diagrams/gp_tree_primer.drawio`**).

**Параметры популяции (`GpConfig`).** Базовые значения включают **`MaxDepth`**, **`PopulationSize`**, **`Generations`**, **`Restarts`**, **`MutationRate`**, **`CrossoverRate`**, **`TerminalConstProbability`**, штраф **`ComplexityPenalty`** за число узлов, **`InvalidPredictionPenalty`** за долю строк с **NaN/∞** в предсказании, **`MaxExpArg`**, **`MaxPowAbs`**, набор **`AllowedOps`**. Именно эта структура подвергается **слиянию** с JSON-переопределениями.

**Внешний цикл `Evolve()`.** Данные разбиваются на обучение и **валидацию** **`SplitTrainValidation(0.2, seed: 42)`**: около **20 %** строк откладываются для оценки **RMSE** кандидата после эволюции. Выполняется **`config.Restarts`** независимых запусков; генератор **`Random`** инициализируется **`12345 + restart`**. В каждом рестарте: **`RunSingleEvolution`** на **обучающей** части → **`OptimizeConstants`** (покоординатный перебор сжатия шага для листьев **`const`**) → выбор кандидата с **минимальным валидационным RMSE** (при равенстве — с лучшим обучающим fitness). Возвращаемое **`bestFitness`** — значение целевой функции на **train** для **выбранного** дерева (после дожима констант). Блок-схема внешнего цикла — **рис. 3.2** (**`diplom/diagrams/fig_3_2_evolve_flow.drawio`**).

**Поколение: оценка, элитизм, размножение.** На каждом шаге для всей популяции считается **`CalculateFitnessOnDataset`**. Популяция сортируется по возрастанию fitness; в следующее поколение **без изменений** копируются **два лучших** дерева (**элитизм**). Остальные места заполняются потомками: два родителя выбираются **`TournamentSelection`** (турнир **k = 3**, победитель с **минимальным** fitness), **`Crossover`** с вероятностью **`crossoverRate`** подменяет случайное поддерево в клоне первого родителя фрагментом второго, иначе потомок — копия первого. Далее к потомку применяется **`Mutate`** (глобальная вероятность **`mutationRate`**, подстройка констант или замена операций/поддеревьев в пределах **`AllowedOps`**). Один шаг «поколения» в обобщённом виде — **рис. 1.2** (**`diplom/diagrams/gp_generation_loop.drawio`**); турнир **k = 3** — **рис. 1.3** (**`diplom/diagrams/gp_tournament_selection.drawio`**, см. также п. 1.2 в гл. 1).

**Целевая функция на обучении.** Для дерева **T** на выборке из **N** строк:  
**fitness = MSE_valid + λ_c · |T| + λ_i · (N_invalid / N)**, где **MSE_valid** — средняя квадратичная ошибка только по **выровненным** предсказаниям, **|T|** — число узлов, **λ_c = ComplexityPenalty**, **λ_i = InvalidPredictionPenalty**, **N_invalid** — число строк, где **`Evaluate`** дал **NaN** или бесконечность. Таким образом, **fitness** одновременно отражает точность, сложность и «долю бессмыслицы» на обучении (см. определение **fitness** в п. 1.2 и п. 1.3).

**Стартовая популяция и тригонометрический seed.** Если аналитическая подгонка синуса/косинуса (`TryFitSinusoid`) дала удовлетворительную относительную ошибку (**`< 0.3`** от размаха **y**), часть особей (**до 1/10 популяции, не более 20**) **клонируется** из аналитического «сида» **`CreateSinusoidSeed()`**. Это ускоряет выход на структуры вида **sin(w·φ(x))**, **cos(·)** и композиции с ними (включая случаи, когда эвристический **тип данных** остаётся **`general`**).

**Связь с материалом главы 1.** Турнирный отбор и цикл «оценка → отбор → кроссовер → мутация» соответствуют блок-схемам генетического программирования (п. 1.2); различие **обучающей**, **валидационной** (внутри **`Evolve`**) и **hold-out-теста** (в многомерном **`ProcessMultipleRegression`**) нужно последовательно отражать в тексте записки и при интерпретации **`result.txt`**, чтобы не смешать показатели.
